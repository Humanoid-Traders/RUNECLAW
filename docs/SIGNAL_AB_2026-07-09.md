# Entry & Exit Gate A/B (2026-07-09)

Two A/Bs on the frozen `alts_1h` benchmark this session — one entry-side (RSI
no-trade zone), one exit-side (partial-TP scale-out). **Both rejected.** Neither
is a shippable edge; both dissolve into artifacts on an already-heavily-optimized
benchmark. Recorded so the levers aren't re-tried.

---

# Part 1 — Entry: Counter-Trend / RSI No-Trade Zone

## Motivation

The live account showed losing **counter-trend oversold shorts** (the bot
shorting into oversold bounces). Hypothesis: the RSI hard-block that rejects
`SHORT` into oversold / `LONG` into overbought is too permissive — widening the
no-trade zone should cut those losers.

Current defaults (`AnalyzerConfig`):

- `RSI_OVERBOUGHT_BLOCK = 72.0` — reject LONG when RSI ≥ 72
- `RSI_OVERSOLD_BLOCK   = 28.0` — reject SHORT when RSI ≤ 28

## Method

Frozen benchmark `data/benchmark/alts_1h` (the alt universe the bot actually
trades), `--honest` fee model, `--fill-mode next_open`. Baseline vs two wider
no-trade zones, RSI thresholds overridden via env at process start.

## Result — REJECTED (regression)

| run | trades | win% | PF | return | Sharpe |
|-----|-------:|-----:|----:|-------:|-------:|
| **baseline 28/72** | **33** | **70%** | **2.44** | **+3.86%** | **0.29** |
| var 32/68 | 14 | 57% | 0.92 | −0.15% | −1.26 |
| var 35/65 | 14 | 57% | 1.00 | ~0% | −1.82 |

Widening the zone removed **19 of 33 trades (−58%)** and collapsed profit factor
from **2.44 → ~1.0**, flipping a profitable run to break-even/negative.

## Why it backfires

The blunt RSI threshold disproportionately kills **winners**, not the
counter-trend losers:

- On the benchmark the biggest edge is **with-trend shorts in downtrends**
  (`TREND_DOWN`: 26 tr, PF 2.84). In a downtrend RSI sits low (≈28–40), so those
  profitable shorts enter *inside* the widened no-short band and get blocked.
- The benchmark already contains **zero counter-trend trades** (attribution:
  `with-trend 30 tr PF 2.64`, `neutral 3 tr`, `counter-trend 0`). Counter-trend
  entries are already filtered — by the **direction-aware MTF-alignment gate**
  (`MTF_ALIGNMENT_GATE_ENABLED`, default ON), which rejects SHORT when the
  higher-timeframe trend is bullish. That is the correct, trend-aware tool;
  a flat RSI threshold is not.

## Conclusion

- **Do not** widen `RSI_OVERSOLD_BLOCK` / `RSI_OVERBOUGHT_BLOCK`. The 28/72
  defaults are near-optimal on this universe.
- Counter-trend protection is already handled by the MTF-alignment gate; the
  RSI block is a coarse backstop for *deeply* over-extended entries only.
- The live counter-trend losses correlate with the **LLM brain being offline**
  (free-tier quota exhausted → rule-engine fallback, weaker entries) rather than
  a missing entry gate. That failure mode is now surfaced by the LLM-degraded
  proactive alert (PR #327) and `/llmstatus` (PR #330). The durable fix is the
  off-code account action: **add a paid LLM key** so the analyzer stops running
  brain-dead.

_No code change ships from this A/B — it is a validated negative result recorded
so the RSI-threshold lever is not re-tried._

# Part 2 — Exit: Partial-TP Scale-Out

## Motivation

Baseline realized **R:R is only 1.19** and avg win (\$28.42) ≈ avg loss (\$26.75)
— winners are cut nearly as short as losers; the 70% hit rate carries the edge.
Partial-TP is already ON (`PARTIAL_TP_ENABLED=True`): bank 50% @ 1.5R, 30% @
2.5R, 20% runner. Hypothesis: shift the scale-out to **let more ride** (bank less
early, bigger runner) to lift R:R without cutting trade count.

## Method

Same benchmark / `--honest` / `next_open`. Env overrides of the partial-TP
close-percentages and TP2 R-multiple.

## Result — REJECTED (in-sample-only, path-dependent)

| run | trades | win% | PF | return | R:R |
|-----|-------:|-----:|----:|-------:|----:|
| **baseline (50/30/20, TP2 2.5R)** | 33 | 70% | 2.44 | +3.86% | 1.19 |
| ride (40/30/30) | 29 | 69% | **5.32** | +4.32% | **1.09** |
| TP2 @ 3.5R | 15 | 60% | 1.94 | +1.26% | 1.20 |

The "ride 40/30/30" PF looks great (2.44 → 5.32) — but it is an **artifact, not a
causal exit edge**:

- Trade count *changed* (33 → 29) and per-trade **R:R actually fell** (1.19 →
  1.09). A pure exit-parameter change should not move the number of entries; the
  gain came from *different trades being taken*, not from better exits.
- Mechanism: the stateful **circuit-breaker / `LIVE_PERF_GOVERNOR` /
  loss-streak latch** are path-dependent on cumulative PnL. A different early
  partial-exit shifts cumulative PnL → trips/untrips the governor at a different
  point → a different subset of later trades is allowed. On a 29–33 trade sample
  that is overfitting to the stateful path, not a real edge.
- OOS validation on `alts_1h_v2` could not cleanly confirm it: the run is
  dominated by the circuit-breaker halting trading (`CIRCUIT_BREAKER: system
  halted due to prior losses`), so there is **no out-of-sample support**.
- `TP2 @ 3.5R` is a clear reject (halves trades, lower everything).

## Conclusion

- **Do not** change the partial-TP close-percentages or TP2 R-multiple. Keep
  50/30/20 @ 1.5R/2.5R. The exit path is already optimized (see prior rounds:
  exit reward/risk, trailing-stop, SL/TP ATR-mults A/Bs).
- Per repo discipline (OOS-validate before shipping), an in-sample-only,
  path-dependent, R:R-negative result does not ship.

# Overall takeaway

Both the entry (RSI) and exit (partial-TP) levers are settled — the `alts_1h`
benchmark is at PF 2.44 and has been optimized across many prior A/B rounds, so
marginal parameter changes now produce **path-dependent noise** that looks like
signal in-sample and vanishes (or reverses) out-of-sample. The live account's
losses are not an entry/exit-tuning problem; they trace to the **LLM brain
running offline** (quota-exhausted → rule-engine fallback), now surfaced by
PR #327 / #330. The durable lever is the off-code account action: **add a paid
LLM key.**
