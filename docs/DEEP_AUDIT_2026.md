# RUNECLAW Audit Synthesis Report

## 1. Overall Verdict

RUNECLAW is structurally sound on the hard-money path — the risk gate, sizing clamp, and exchange-side SL/TP placement work — but the audit surfaces a consistent pattern of **sophisticated subsystems that are wired in name only**. Three categories dominate: (a) **adaptive/learning machinery that is silently inert** (confidence calibrator trained entirely on confidence=0.0, setup-expectancy nudge永远 zero from a regime-key mismatch, strategy_router computed-then-discarded, walk-forward sweeping a no-op parameter), (b) **live-data freshness gaps** (REST `last` drives SL/TP and trailing stops with no staleness check; the WS heartbeat trusts socket state, not data flow, so a silently-stalled feed never trips PAUSED), and (c) **default-OFF safety/quality features** the runbook tells operators to enable but which are dead, mis-wired, or unobservable when flipped on. None of these actively lose money in the **default** configuration, but several create a dangerous gap between what operators *believe* is protecting them and what actually runs — most acutely the staleness-blind live SL/TP path and the calibration learner that would corrupt the trade gate the moment it's enabled per the documented runbook.

### Cross-domain root-cause corroboration

- **REST/WS price freshness** is the single most-corroborated theme: #1, #3, #16, #45 (and #46) across the execution, sltp, and data-ws domains all trace to "liveness/price trusted without a freshness check." Fix the contract once, apply everywhere.

- **Default-OFF / dead adaptive features** recur across nearly every domain (#4, #5, #6, #7, #8, #21, #57): the bot ships substantial intelligence that is inert by default or unwired — the highest-leverage *product* work is deciding, per feature, "wire it (with backtest parity + observability) or delete it," because the current middle state misleads operators about what's protecting them.
